## 핵심 차이점

1. 고객용 앱은 주요 화면 흐름이 구현되어 있으나, 백엔드 API는 교환 신청/사진 분석/예약/최종 검수 Mock 일부만 연결되어 있다.
2. 2차 산출물은 고객, 수거 크루, 운영 관리자를 하나의 서비스 흐름으로 정의하지만 현재 백엔드는 고객 흐름 중심이다.
3. 크루 앱은 가입/로그인/대시보드 UI는 있으나 백엔드의 콜 목록, 콜 수락, 출발/도착/완료, 위치 업데이트 API와 연결되어 있지 않다.
4. 관리자 유스케이스인 수거 신청/상태 관리, 최종 검수, 재검수 처리, 보상/정책 관리, 조직/제휴 관리는 API가 없다.
5. DB 객체정의서 기준의 12개 객체 중 `pickup_requests`, `tracking_events`, `re_reviews`, `credits`, `market_products`, `pickup_result_reports`, `notifications`가 코드 구조에 명확히 드러나지 않는다.
6. 기존 백엔드 응답 문구 일부가 깨져 있어 고객 앱에서 상태/근거/결과 문구를 안정적으로 표시하기 어렵다.

## 1차 반영 범위

- 기존 고객 앱 API 호환 유지
- 고객 API 확장: 예상 보상가 확인, 바로 콜, 트래킹, 재검수, 크레딧, 알림, 마켓
- 크루 API 추가: 콜 목록 조회, 콜 수락, 출발, 위치 업데이트, 도착, 수거 완료
- 관리자 API 추가: 수거 신청 조회, 최종 검수 등록, 재검수 처리
- Mock 기반 인메모리 저장소에서 2차 객체정의서의 핵심 객체가 드러나도록 상태 모델 확장
